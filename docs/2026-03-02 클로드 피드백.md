## 1. 코드 구조 및 유지보수성 — 5 / 10

**근거:**

- **단일 파일 구조**: 1183줄로 관리 가능한 규모이긴 하나, HTML/CSS/JS가 혼재되어 있어 특정 로직을 찾는 데 시간이 걸림. GitHub Pages 배포 단순화라는 명확한 이유가 있어 의도적 트레이드오프로 볼 수 있음
- **함수 분리**: `updateAll()`, `updateList()`, `updateChart()` 등 기능별 함수 분리는 잘 되어 있음. 다만 Firebase 모듈 스크립트와 일반 스크립트 간 `window.*` 브릿지 패턴이 결합도를 높임
- **전역 변수 의존**: `items`, `dividendRecords`, `selectedCategories`, `editingIndex`, `editingDividendId` 등 모든 상태가 전역 변수. 어디서든 직접 변경 가능하여 사이드이펙트 추적이 어려움
- **코드 중복**: `downloadCSV()`(index.html:1060)와 `window.downloadChartCSV()`(index.html:546)가 거의 동일한 CSV 생성 로직을 반복. JSON 다운로드도 마찬가지

**개선 제안:**

- CSV/JSON 생성 로직을 공통 함수로 추출 (`generateCSV(items, dividends)`)
- 상태 객체를 하나로 묶기 (`const state = { items, dividendRecords, ... }`)

## 2. 코드 품질 및 안정성 — 4 / 10

**근거:**

- **에러 처리 부족**: Firebase 작업에서 `saveToFirebase()`(index.html:482)에 try-catch가 없음. `loadChartClick()`(index.html:523)도 네트워크 실패 시 무방비. `deleteChartClick()`만 try-catch 존재
- **입력 검증 미흡**: `submitItem()`(index.html:703)에서 이름과 숫자만 검증하고, 카테고리가 0개인 경우나 음수 가격은 허용됨. `amount`에 0이나 음수 입력 가능
- **XSS 취약점**: `innerHTML`에 사용자 입력값을 직접 삽입하는 패턴이 다수 존재
  - index.html:508 — `${chartName}`이 이스케이프 없이 HTML에 삽입
  - index.html:760 — `${i.name}` 직접 삽입
  - index.html:639 — 카테고리명이 onclick 핸들러 문자열에 직접 삽입 — 작은따옴표가 포함된 카테고리명으로 코드 인젝션 가능
- **Firebase 설정 노출**: 프론트엔드 전용이므로 구조적으로 불가피하나, Firestore Security Rules가 적절히 설정되어 있는지가 실질적 보안 핵심 (코드에서는 확인 불가)

**개선 제안:**

- `innerHTML` 대신 `textContent` + `createElement` 조합 사용 또는 최소한 이스케이프 함수 도입
- 모든 Firebase 비동기 호출에 try-catch + 사용자 알림 추가
- 입력값 범위 검증 강화 (양수 체크, 최대값 등)

## 5. 반응형 및 접근성 — 3 / 10

**근거:**

- **반응형**: `<meta name="viewport">` 태그 자체가 없음! 모바일에서 데스크탑 크기로 렌더링될 가능성 높음. 미디어쿼리도 전혀 없음. `max-width: 700px`만으로는 모바일 대응 불충분
- **터치 인터랙션**: 버튼 크기(`min-height: 35px`)는 모바일에서 최소 터치 타겟(44px)에 미달. 액션 버튼은 `padding: 4px 8px`로 매우 작음
- **키보드 내비게이션**: 새 카테고리 Enter 키 지원(index.html:1176)은 좋으나, 모달에 포커스 트랩이 없어 Tab 키로 모달 뒤 요소에 접근 가능
- **스크린리더**: `<label>`이 `for` 속성 없이 사용되는 곳이 있고, 이모지가 스크린리더에서 장황하게 읽힘. `aria` 속성 전무. 모달의 `role="dialog"` 미적용

**개선 제안:**

- `<meta name="viewport" content="width=device-width, initial-scale=1.0">` 필수 추가
- 모바일 미디어쿼리 추가 (`@media (max-width: 600px)`)
- 모달에 `role="dialog"`, `aria-modal="true"`, 포커스 트랩 추가
- 버튼 최소 터치 타겟 44px 확보

## 6. 성능 — 6 / 10

**근거:**

- **초기 로딩**: CDN 3개(Chart.js, datalabels 플러그인, Firebase SDK 3모듈) + Google Fonts. 번들러가 없어 트리셰이킹 불가하나, 전체 앱이 ~43KB HTML 하나이므로 실제 로딩은 빠름
- **렌더링**: `updateAll()`이 항상 6개 함수를 모두 호출하여 전체 리렌더링. 종목 1개 삭제해도 차트 재생성 + 전체 리스트 innerHTML 교체. 데이터가 적을 때는 문제없으나 비효율적 구조
- **차트 리렌더링**: `updateChart()`가 매번 `portfolioChart.destroy()` → `new Chart()` 수행(index.html:959-999). Chart.js의 `update()` 메서드를 쓰면 더 효율적
- **확장성**: 종목 수백 개 수준에서는 DOM 전체 교체 방식이 눈에 띄게 느려질 수 있음

**개선 제안:**

- Chart.js `update()` 메서드로 데이터만 갱신
- `<link rel="preconnect">` 추가로 CDN 연결 최적화
- 종목 수 증가 대비 가상 스크롤 또는 페이징 고려

### 강점

- **기능 완성도가 높음**: 개인 프로젝트로서 CRUD + 차트 + 다중 저장소 + 다크모드 + CSV/JSON 내보내기까지 실용적인 기능 세트 구현

- **실제로 사용 가능한 수준**의 완성된 앱

### 약점

- viewport 미설정으로 **모바일 사용이 사실상 불가**
- **보안 취약점(XSS)**이 여러 곳에 존재
